# Raw Session: NovaLaunch Q3 Campaign

## Session 1: First proposal review

User: We need to revise the NovaLaunch Q3 campaign proposal. The first version looked big, but the Head of Growth said it did not explain why this would convert.

Agent: What was the first campaign direction?

User: It leaned heavily on KOL launch visibility. Big creator names, social buzz, short launch burst. It looked presentable, but the conversion path was weak.

Agent: Should we keep KOL as the main plan and add conversion logic?

User: I do not think so. The budget is not enough for a real KOL-heavy launch. If we put most of the money there, we will only get surface visibility and then still fail to prove conversion.

Agent: What should replace it?

User: The stronger direction is user-generated proof. We seed real use cases, collect lightweight user stories, show conversion path clarity, and use small creator seeding only to amplify proof. The main plan should not be KOL-led.

Agent: So the key decision is to move away from KOL-heavy execution toward a user-proof loop.

User: Yes. The proof object is not "we can make a big splash." The proof object is "this campaign can create credible usage proof and move users through a clearer conversion path under constrained budget."

Agent: What should be rejected explicitly?

User: Reject KOL-heavy launch as the main plan. Also reject a broad awareness-first plan without conversion proof. The revised deck needs to explain why this is a smarter use of budget.

## Session 2: Later stakeholder input

User: New input. One stakeholder asked whether we should reconsider KOL because executives understand KOL better and it feels more visible.

Agent: Does that mean the strategy is changing back to KOL-led?

User: No, not yet. This is only a request to revisit. We need to ask what new evidence has changed. If there is no new budget or channel data, I do not want the next agent to revive KOL as if it is confirmed.

Agent: How should the next proposal handle it?

User: It can mention a bounded KOL test if it supports the proof loop, but the main direction should stay user-generated proof and conversion clarity. The question is how to respond to stakeholder concern without losing the logic.

Agent: What is the next concrete task?

User: Prepare a revision plan for the next proposal deck. It should show the main argument, what changed from the first version, how to handle the stakeholder's KOL question, and what evidence would be needed to reopen KOL as a major path.
