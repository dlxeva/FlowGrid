# Raw Session: Community Activation Sprint

## Session 1: Mechanism review

User: We need to revise the Community Activation Sprint mechanism. The first draft had points, badges, weekly ranking, bonus tasks, and multiple participation tracks.

Agent: That sounds engaging. Should we keep the full gamified structure?

User: No. The operations review showed it is too hard to execute. Frontline moderators cannot explain all the rules consistently, and settlement disputes will eat the team alive.

Agent: What specifically is too complex?

User: Points are hard because users will argue about what counts. Badges need tracking. Weekly ranking creates disputes. Bonus tasks add edge cases. Multiple tracks make the whole thing impossible to communicate in one screen.

Agent: What should replace it?

User: A lightweight loop. One core task, one proof submission, one weekly showcase, one simple reward rule. It should be easy for users to understand and easy for moderators to operate.

Agent: So the key decision is to simplify the mechanism.

User: Yes. The proof object is not "the activity has many mechanics." The proof object is "the mechanism can drive participation without creating execution overload."

Agent: What should be explicitly rejected?

User: Reject multi-track tasks, ranking as the main incentive, badge economy, bonus task stack, and complex point settlement.

## Session 2: Stakeholder engagement pressure

User: New input. A stakeholder asked whether we should bring points and ranking back because they look more engaging.

Agent: Should we restore points and ranking?

User: Not as a default. This is a concern about engagement, not proof that complexity is acceptable. We need evidence: can moderators explain it, can we settle disputes, does it actually increase quality participation?

Agent: Is there any acceptable way to test it?

User: Maybe a bounded test. For example, one optional weekly spotlight score handled internally, without making points the main user-facing rule. But the core mechanism should stay lightweight.

Agent: What is the next concrete task?

User: Prepare a revised mechanism plan. It should preserve the lightweight loop, answer the engagement concern, and define what evidence would justify reintroducing points or ranking later.
