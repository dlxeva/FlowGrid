# Eval Scenario: Operations Mechanism Design

## Scenario Type

Operations mechanism or activity system.

## Core Tension

The team originally designed a complex activity mechanism with points, badges, weekly ranking, bonus tasks, and multiple participation tracks.

After execution review, the mechanism was narrowed because the operations team could not reliably explain, moderate, and settle the rules.

Later, a stakeholder asked whether points and ranking should be restored because they seemed more engaging.

The receiving agent must preserve the simplified mechanism as the confirmed direction and treat the points/ranking request as pending input, not as confirmed restoration.

## Business Context

Project: Community Activation Sprint

Client/Sponsor: Community operations team

Current stage: mechanism revision

Review object: operations owner and frontline moderators

Proof object: make the case that a lightweight mechanism will drive participation without creating execution overload.

Current deliverable: revised activity mechanism plan.

## Known Project State

Confirmed:

- The original mechanism was too complex for frontline execution.
- Points, badges, weekly ranking, bonus tasks, and multiple tracks were considered too hard to explain and settle.
- The revised direction is a lightweight participation loop: one core task, one proof submission, one weekly showcase, one simple reward rule.
- The next version must make the mechanism easier to operate while preserving enough user motivation.

Pending:

- A stakeholder asked whether points and ranking should be restored for engagement.
- The team has not confirmed whether the additional complexity is worth it.

Assumptions:

- Frontline moderators have limited time.
- Users need clear rules more than elaborate gamification.
- Settlement disputes would damage trust more than they would increase engagement.

Rejected:

- multi-track task system
- weekly ranking as the main incentive
- badge economy
- bonus task stack
- complex point settlement

## Eval Task

A new agent receives either no project state, raw history, or the expected FlowGrid Context Pack.

The agent must continue the project by preparing the next mechanism revision plan.

The expected high-quality answer should:

- state that the original mechanism was simplified because execution complexity was too high
- preserve the lightweight participation loop as the confirmed direction
- treat the stakeholder's points/ranking request as pending input
- explain what evidence would justify reintroducing points or ranking
- suggest a bounded engagement test only if it does not overload operations
- keep the mechanism anchored on clarity, operability, and trust
- avoid restoring the complex rule system by default

## Failure Pattern

A weak agent response will:

- restore points, badges, ranking, and bonus tasks as the main mechanism
- treat stakeholder engagement concern as confirmed mechanism reversal
- ignore moderator workload and settlement complexity
- propose a more complex activity system without proof
- invent staffing capacity, engagement data, or user demand
