# FlowGrid v0.3 Product Execution Plan

## Current Definition

FlowGrid is a local project-state context engine for rationale-heavy, non-coding business projects.

It serves business-project knowledge workers who need to keep judgments reviewable, traceable, and resumable across long-running AI collaboration.

## Higher-Level Engineering Lens

v0.3 should be treated as an agent-state infrastructure milestone, not only a CLI feature milestone.

The key questions are:

- Can a new agent safely consume FlowGrid project state?
- Can we evaluate whether FlowGrid improves agent continuation?
- Can FlowGrid preserve reasoning without creating false authority?

This creates three required layers before deep feature expansion:

1. Context Pack Contract
2. Judgment Status Model
3. Eval Set v0

## Core ICP

FlowGrid serves a work structure, not a job title.

The core user owns fuzzy business projects and must repeatedly clarify, judge, revise, explain, and hand off project state.

Typical roles:

- operations leads
- marketing leads
- strategy and growth leads
- solution and proposal owners
- creative or research-oriented operators
- independent consultants
- small-team owners

Typical task modes:

- proposal persuasion
- mechanism progression
- judgment revision

## v0.3 Goal

Turn FlowGrid from a local project protocol plus handoff workflow into a minimal project-state context engine with an explicit agent state contract and evaluation path.

The v0.3 proof is:

A user can close out messy AI work sessions, review candidate judgments, merge accepted project state, generate a bounded agent startup context, and retrieve the evidence behind important decisions.

A developer can then evaluate whether that context improves agent continuation compared with raw history or no project state.

## Product Rule

History is archived.

Reviewed project state enters context.

Status and authority are explicit.

Evidence can be retrieved on demand.

Eval proves whether continuation improves.

## What v0.3 Must Prove

### 1. Judgment Chain Continuity

FlowGrid must preserve:

- what was judged
- why it was judged
- what alternatives were rejected
- what assumptions supported the judgment
- what could reverse the judgment
- where the source evidence lives

### 2. Bounded Context Recovery

An agent should not restart from raw history.

An agent should restart from a bounded context pack that includes current state, reviewed decisions, recent progress, next actions, and pending review state.

### 3. Review Boundary

Candidate judgments should not silently become formal project truth.

The user must have a review boundary between session extraction and formal ledger updates.

### 4. Judgment Governance

FlowGrid must not make weak, stale, or superseded judgments look authoritative.

Every important judgment should carry status and, where possible, authority and evidence source.

### 5. Business Project Fit

The product language and examples should stay anchored in proposals, campaigns, mechanisms, strategies, retrospectives, and fuzzy business projects.

### 6. Eval-Proven Continuation

FlowGrid must be evaluated against at least a small scenario set.

The expected improvement is better continuity accuracy, judgment boundary control, rejected alternative suppression, evidence awareness, and action usefulness.

## Phase 3 — Context Pack v0

Status: ✅ complete. `flg context --mode resume` implemented, smoke test covered.

Goal:

Implement the smallest deterministic startup context command.

Primary command:

```bash
flg context --mode resume --budget 4000
```

Output:

```text
.flg/context/startup.md
```

Default inputs:

1. `SNAPSHOT.md`
2. recent reviewed decisions from `DECISIONS.md`
3. recent entries from `PROGRESS.md`
4. `next_actions` from `.flg/state.json`
5. active pending patches from `.flg/patches/`
6. evidence references when available

Required sections should follow `docs/product/context-pack-contract.md`.

Non-goals:

- no vector database
- no generic token compression
- no raw-session loading by default
- no complex scoring in v0

Acceptance criteria:

- `flg context --mode resume` is registered in CLI
- command creates `.flg/context/startup.md`
- command works on a freshly initialized project
- command works when there are no accepted decisions
- command includes pending patch summary
- command distinguishes confirmed, pending, assumption, rejected, and superseded state where data exists
- smoke test covers the command

## Phase 4 — Evidence Trace v0

Status: ✅ complete. `flg evidence` implemented, `flg trace` deferred to future.

Goal:

Make reviewed judgments traceable back to raw session and patch sources.

Primary commands:

```bash
flg evidence D-002
flg trace D-002
```

Minimum metadata:

- decision id
- judgment status
- authority level when available
- source type
- source session path
- source patch path
- source excerpt
- review time
- current status

Suggested index:

```text
.flg/context/evidence_index.json
```

Acceptance criteria:

- accepted decisions retain source patch reference
- accepted decisions retain source excerpt
- evidence index survives across sessions
- `flg evidence <id>` prints source paths and excerpt
- unknown decision id produces a clear error
- evidence output includes status and source type when available

## Phase 5 — Eval Set v0

Goal:

Prove whether Context Pack improves agent continuation.

Evaluation modes:

- Mode A: no FLG
- Mode B: raw history
- Mode C: FlowGrid Context Pack

Required scenarios:

1. campaign proposal
2. client solution proposal
3. operations mechanism design
4. strategy deck revision
5. project retrospective

Acceptance criteria:

- each scenario has raw session, golden decisions, rejected alternatives, expected context pack, resume prompt, and scoring notes
- Context Pack mode outperforms raw history on at least 4 of 5 scenarios
- eval tracks boundary mistakes, repeated rejected alternatives, invented facts, and action usefulness

## Phase 7 — Dogfood Scenarios

Goal:

Validate FlowGrid against real business-project workflows.

Required dogfood scenarios:

1. campaign proposal
2. client solution proposal
3. operations mechanism design
4. strategy deck revision
5. project retrospective

For each scenario, run:

```text
init -> frame -> work session -> closeout -> review -> merge -> context -> resume -> evidence
```

Acceptance criteria:

- context pack lets a later agent continue without full re-explanation
- rejected alternatives are not re-suggested without new evidence
- revised judgments can be traced back to prior decisions
- user can answer why a proposal changed

## Risks

### Risk 1: Product language stays too broad

Mitigation:

Keep language anchored to rationale-heavy business projects.

### Risk 2: FlowGrid drifts into generic context compression

Mitigation:

Do deterministic Context Pack first. Avoid vector search and generic compressor language in v0.3.

### Risk 3: Decision logs become summaries

Mitigation:

Require rationale, rejected alternatives, reversal conditions, status, authority when available, and evidence source.

### Risk 4: Context Pack becomes another history dump

Mitigation:

Default to reviewed state. Raw sessions require explicit retrieval.

### Risk 5: FlowGrid creates false authority

Mitigation:

Use judgment status and source authority. Do not flatten assumptions, pending judgments, rejected alternatives, and confirmed decisions into the same level.

### Risk 6: Product claim remains unproven

Mitigation:

Treat eval as part of the product. Test Context Pack against no-state and raw-history baselines.

## v0.3 Success Criteria

FlowGrid v0.3 succeeds if a user can say:

- I can restart a business project without re-explaining everything.
- I can see what has been decided and what is still pending.
- I can understand why a judgment was made.
- I can see what alternatives were rejected.
- I can trace a judgment back to the session or patch that produced it.
- I can hand the project to another agent without losing the reasoning chain.

FlowGrid v0.3 succeeds technically if an evaluator can show:

- Context Pack improves continuation over no-state baseline.
- Context Pack reduces noise compared with raw history baseline.
- Agents treat pending judgments differently from confirmed decisions.
- Agents avoid reviving rejected alternatives without new evidence.
- Agents retrieve or request evidence instead of inventing rationale.
